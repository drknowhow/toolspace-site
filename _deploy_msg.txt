deploy: GitHub Actions → Netlify (prod) on push to main

Mirrors yepgent.com's deploy pattern: GH Actions runs `netlify
deploy --prod` with NETLIFY_AUTH_TOKEN + NETLIFY_SITE_ID from repo
secrets. Auth-isolated and leak-safe (no Netlify GitHub App OAuth
required, and the token never lands in a public log).

The Netlify-side repo connection still exists from site creation
but its webhook can't pull the repo (GH App not authorized for
drknowhow/toolspace-site). GH Actions sidesteps that — pushes the
build artefact directly via the Netlify CLI.

Concurrency: a single deploy-prod queue prevents two pushes from
racing into prod. Out-of-order builds aren't catastrophic for a
static site, but ordered deploys make rollbacks cleaner.

.netlifyignore keeps CI/scratch files off the CDN.
